# Publishing FAAM (hosted backend + thin Mac app)

This is the safe way to ship FAAM so AI **and** payments "just work" for users
without anyone bringing their own key — **your** OpenAI and Stripe keys live only
on your server (in environment variables), never inside the downloadable app.

```
            ┌─────────────────────┐         ┌──────────────────────────────┐
 Downloads  │  FAAM.app (window)  │  HTTPS  │  Your FAAM backend (app.py)  │
 the app →  │  thin client, NO    │ ──────▶ │  holds OpenAI + Stripe keys  │
            │  keys, no server    │         │  enforces tiers + spend caps │
            └─────────────────────┘         └──────────────────────────────┘
```

---

## Step 1 — Deploy the backend

FAAM is **stdlib-only** (no `pip install`), so there's nothing to build.
Any host that runs Python works (Render, Railway, Fly.io, a VPS). Render is the
simplest because `render.yaml` is already here.

### Render (recommended)
1. Push this folder to a **GitHub** repo.
2. Render → **New → Blueprint** → pick the repo (it reads `render.yaml`).
3. Set these env vars in the dashboard (the blueprint marks them `sync:false`):
   | Variable | Value |
   |---|---|
   | `OPENAI_API_KEY` | your OpenAI key (`sk-...`) |
   | `STRIPE_SECRET_KEY` | your Stripe secret key (`sk_test_…` to start) |
   | `FAAM_BASE_URL` | `https://<your-render-url>` (the app's own public URL) |
   | `FAAM_SESSION_SECRET` | auto-generated by the blueprint (leave it) |
   | `FAAM_DATA_DIR` | `/var/data` (already set; the persistent disk) |
4. Deploy. Render gives you `https://faam-xxxx.onrender.com`.
5. Put that exact URL in `FAAM_BASE_URL` and redeploy (Stripe/Google redirects use it).

> **Persistence:** accounts + usage live in `FAAM_DATA_DIR`. The blueprint mounts a
> 1 GB disk at `/var/data` so they survive deploys. Without a disk, the filesystem
> is wiped on every deploy and everyone gets logged out. (For real scale, move to a
> database later — JSON files are fine for a launch.)

### Any other host
Set the same env vars and run `python3 app.py`. The server binds to `$PORT`
(provided by the host) automatically.

## Step 2 — Verify the backend
- Visit `https://<your-url>/api/health` → should say `ai_enabled: true`.
- Visit `https://<your-url>/login`, sign in as **dev** (your admin) → dashboard loads.
- Open **Upgrade** → pick a plan → you should reach Stripe Checkout. Use test card
  `4242 4242 4242 4242` while on a `sk_test_…` key. After paying you're redirected
  back to `FAAM_BASE_URL` and the tier unlocks.

## Step 3 — Point the Mac app at your backend (make it a thin client)
1. Open `faamview.swift` and set:
   ```swift
   let bakedRemoteURL = "https://<your-url>"
   ```
   With this set, the app **skips the local server and the API-key prompt** — it just
   opens a window onto your hosted FAAM. (Empty = local/dev mode, unchanged.)
2. Compile a universal, **signed** binary (see Step 4) and drop it at
   `FAAM.app/Contents/MacOS/faamview`.
3. Tip: you can test remote mode before baking it in:
   `FAAM_REMOTE_URL=https://<your-url> ./FAAM.app/Contents/MacOS/faamview`

---

## Step 4 — Sign, notarize, and distribute the Mac app
Needed so users don't hit Gatekeeper's "unidentified developer" wall.
1. Enroll in the **Apple Developer Program** ($99/yr) → get a *Developer ID Application* cert.
2. Ship a **pre-compiled** `faamview` (don't rely on each user having Xcode):
   ```sh
   swiftc -O -target arm64-apple-macos11  -framework Cocoa -framework WebKit faamview.swift -o faamview-arm64
   swiftc -O -target x86_64-apple-macos11 -framework Cocoa -framework WebKit faamview.swift -o faamview-x86_64
   lipo -create faamview-arm64 faamview-x86_64 -output FAAM.app/Contents/MacOS/faamview
   ```
3. Sign + notarize + staple:
   ```sh
   codesign --deep --force --options runtime --sign "Developer ID Application: YOUR NAME (TEAMID)" FAAM.app
   ditto -c -k --keepParent FAAM.app FAAM.zip
   xcrun notarytool submit FAAM.zip --apple-id you@email.com --team-id TEAMID --password APP_SPECIFIC_PW --wait
   xcrun stapler staple FAAM.app
   ```
4. Re-zip (or build a `.dmg`) and host it on your download page.

## Going-live checklist
- [ ] Stripe: swap `sk_test_…` → **live** `sk_live_…`; do a real $ test; consider Stripe Tax.
- [ ] Change the **dev** admin password (it ships with a known hash in `seed_users`).
- [ ] Tune the hidden spend caps in `TIER_CAP_CENTS` to real monthly budgets vs. each
      plan's price (current values are test-sized: Lite 2¢ / Pro 10¢ / Max $1).
- [ ] Add **Terms** + **Privacy** pages; keep the "information, not financial advice" line.
- [ ] (Optional) Google sign-in: create OAuth creds in Google Cloud Console and add the
      authorized redirect `https://<your-url>/auth/google/callback`.

## Never ship secrets
- The **`FAAM-PRIVATE-with-key.zip`** has a baked-in key — that's a personal backup,
  **never** put it on your download page.
- Keys belong in **server env vars only**. The distributed app should contain *no*
  `sk-...` or `sk_live_...` anywhere.
